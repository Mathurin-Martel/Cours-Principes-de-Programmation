Une image contient le programme ainsi que l'environnement pouvant le faire tourner.

Terminal :
docker images : Listes les images sous forme de tableaux avec les colonnes suivantes : IMAGE, ID, DISK USAGE, CONTENT SIZE, EXTRA
docker ps : Listes les conteneurs en cours d'éxécution sous forme de tableaux avec les colonnes suivantes : CONTAINER ID, IMAGE, COMMAND, CREATED, STATUS, PORTS, NAMES
docker ps -a : Liste tout les conteneurs, ceux encore en cours d'éxécution et ceux qui ont fini leur éxécution.
docker run -it : run c'est pour exécuter. -it c'est pour que ce soit en mode interactive terminal
docker run hello-world : Lors du premier lancement, docker ne connait pas l'image hello-world. Il va donc la télécharger depuis Internet. Il lance ensuite l'image hello-world. Au deuxième lancement, Docker n'a plus besoin de télécharger et exécute directement.
Lorsqu'on exécute docker images une ligne dont la première valeur est hello-world:latest a été ajoutée.
Le ":latest" signifie qu'au moment du téléchargement, c'était la dernière version de l'image.

"docker run -it alpine:latest /bin/sh" lance un environnement linux minimal. On peut lancer ps mais pas nano par exemple.
Pour installer curl dessus on lance apk add --no --cache curl.

Si on exécute "exit" puis qu'on exécute de nouveau "docker run -it alpine:latest /bin/sh", on peut se rendre compte que curl, ainsi que toutes les autres modifications qu'on aurait apportées, ont disparu.
C'est parce qu'on a créé un nouveau conteneur a partir de l'image
Pour sauvegarder les modifications il faut lancer deux terminaux. 
Sur le premier, on lance l'image avec docker run -it mon_image /bin/sh.
Sur le second, on lance docker export ID_DU_CONTENEUR_EN_COURS_D'EXECUTION -o QUELQUE_CHOSE.tar
On peut désormais lancer "exit" sur le premier terminal et "docker import QUELQUE_CHOSE.tar NOM_DE_L'IMAGE:VERSION_DE_L'IMAGE" (on peut mettre V1 par exemple).

docker rm ID_OU_NOM_DU_CONTENEUR supprime le conteneur.
docker rmi ID_OU_NOM_DE_L'IMAGE supprime l'image dont on a donné l'ID MAIS seulement si les conteneurs se servant de l'image ont été supprimés avant.

docker run -d -p 8080:80 [--name monServeur] nginx. Lance une page internet sur http://localhost:8080/.
[] veut dire optionnel, il faut les enlever quand on veut exécuter la commande.
-d : détaché
-p : port
8080:80 = ma_machine:conteneur

docker exec -it monServeur bash sert à lancer le conteneur où se trouve index.html
Pour installer nano dessus, impossible de faire apk add --no -cache nano car il ne connait pas apk. Il faut faire apt update puis apt install nano.

Si on fait exit sur ce conteneur ça ne stoppe pas la page internet. Il faut faire docker stop NOM_DU_CONTENEUR.
Si on relance le conteneur, les modifications restent.

La raison c'est que quand on run une image ça crée un NOUVEAU conteneur à partir de cette image mais si on run un CONTENEUR alors ça garde les modifications.

Docker propose 3 types principaux :
A.Bind mounts (montages liés)
Ils relient un dossier de l'hôte à un dossier dans le conteneur.
On voit les fichiers en temps réel.
Dépend du système de fichiers hôte.
Risque de sécurité si mal configuré.
docker run -v source:destination image
source est un chemin de la machine hôte
destination se trouve dans le conteneur

B..Volumes gérés par Docker
Ce sont les volumes recommandés.
Stockés dans /var/lib/docker/volumes/
Indépendants du système hôte.
Plus sûrs et facilement sauvegardables.
docker volume create monvolume
docker run -v monvolume:/chemin/image

C.tmpfs (mémoire uniquement -Linux)
Stockage temporaire uniquement en RAM
Rapidité
Aucune trace sur le disque
Données perdues au redémarrage
docker run -tmpfs /chemin image

docker volume ls : Liste les volumes
docker volume inspect monvolume : Affiche un json contenant des informations sur monvolume
docker volume rm monvolume : : Efface monvolume

docker run -it -v monvolume:/data alpine sh
Faites echo "hello" > /data/test.txt
Puis docker rm -f id_du_conteneur
Puis docker run -it -v monvolume:/data alpine sh
Vous verrez que test.txt existe toujours.

Partie Bind mounts : Monter un dossier de l'hôte
docker run -it -v /home/user/projets:/app alpin sh
Tout changement dans /home/user/projets se reflète dans /app
Exemple concret : Faire persister une base de données SQL
docker volume create mysql_data
docker run -d-name mysql -e MYSQL_ROOT_PASSWORD=pass-b mysql_data:/var/lib/mysql mysqsl:8

Gestion des réseaux avec Docker
docker network 

Le réseau bridge est le réseau par défaut
docker run -d --name web nginx
Docker le met automatiquement dans bridge
docker network inspect bridge

Le réseau host :
Le conteneur utilise le réseau directement du host
Pas d'IP Docker, pas d'isolation
docker run --network host nginx
Ça ne fonctionne pas sur toutes les machies
Pour quelle raison?

Le réseau none :
Le conteneur ne peut communiquer avec aucun autre conteneur
Aucun réseau, aucun accès
docker run --network none -it alpine sh
Lancer un ping www.google.com à l'intérieur du conteneur. Que remarquez-vous ? Que ça ne marche pas. Le conteneur ne peut, par Internet, recevoir de données de l'extérieur ou en envoyer.

Le réseau personnalisé :
Créer votre propre réseau.
docker run -d --name site --network monreseau nginx
docker run -d --name api --network monreseau alpine sleep 1000
tester la communication avec docker exec -it site ping api
(Ils se trouvent automatiquement grâce au DNS interne de Docker)

Connecter et déconnecter un conteneur : 
Ajouter un conteneur à un réseau existant : docker network connect monreseau site
Le retirer : docker network disconnect monreseau site
